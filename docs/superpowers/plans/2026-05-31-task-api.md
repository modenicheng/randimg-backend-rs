# Task Management RESTful API Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add RESTful API for task management — trigger crawl jobs with parameters, enhance task listing with total count, cancel pending tasks, retry failed tasks.

**Architecture:** Extend `src/handlers/task.rs` with POST /tasks/crawl for job creation, add total count to existing `list_tasks`, add cancel (DELETE) and retry (POST) endpoints. Cancel/retry use direct SQL on Apalis `Jobs` table since Apalis doesn't expose public cancel/retry APIs. All endpoints require `AuthUser`.

**Tech Stack:** Axum, sqlx (raw SQL on Apalis pool), serde

---

## API Design

| Method | Path | Handler | Description |
|--------|------|---------|-------------|
| `POST` | `/tasks/crawl` | `create_crawl` | Submit a crawl job with parameters (NEW) |
| `GET` | `/tasks` | `list_tasks` | List all tasks with `?task_type=&status=` filters (EXISTING, enhanced with total count) |
| `GET` | `/tasks/{task_id}` | `get_task` | Get single task (EXISTING, unchanged) |
| `DELETE` | `/tasks/{task_id}` | `cancel_task` | Cancel a pending task (NEW) |
| `POST` | `/tasks/{task_id}/retry` | `retry_task` | Retry a failed/killed task (EXISTING handler, rewrite to use Apalis table) |

**Note on cancel/retry:** Apalis v1.0.0-rc.9 doesn't expose public cancel/retry methods on SqliteStorage. These endpoints use direct SQL on the `Jobs` table. This is safe for SQLite but should be revisited if Apalis adds a public API.

### Task 3: Add cancel_task and rewrite retry_task

**Files:**
- Modify: `src/handlers/task.rs`

- [ ] **Step 1: Add cancel_task handler**

```rust
/// Cancel a pending task by setting its status to Killed.
/// Uses direct SQL since Apalis doesn't expose a public cancel API.
async fn cancel_task(
    State(state): State<Arc<AppState>>,
    _auth: AuthUser,
    Path(task_id): Path<String>,
) -> Result<Json<serde_json::Value>, AppError> {
    let result =
        sqlx::query("UPDATE Jobs SET status = 'Killed' WHERE id = ?1 AND status = 'Pending'")
            .bind(&task_id)
            .execute(&*state.apalis_pool)
            .await
            .map_err(|e| AppError::Internal(e.to_string()))?;

    if result.rows_affected() == 0 {
        return Err(AppError::NotFound(
            "Task not found or not in pending status".into(),
        ));
    }

    Ok(Json(serde_json::json!({
        "message": "Task cancelled",
        "task_id": task_id,
    })))
}
```

- [ ] **Step 2: Rewrite retry_task to use Apalis Jobs table**

Replace the existing `retry_task` which queries the old `background_tasks` table:

```rust
/// Retry a failed or killed task by resetting its status to Pending.
/// Uses direct SQL since Apalis doesn't expose a public retry API.
async fn retry_task(
    State(state): State<Arc<AppState>>,
    _auth: AuthUser,
    Path(task_id): Path<String>,
) -> Result<Json<serde_json::Value>, AppError> {
    let result = sqlx::query(
        "UPDATE Jobs SET status = 'Pending', attempts = 0 \
         WHERE id = ?1 AND status IN ('Failed', 'Killed')",
    )
    .bind(&task_id)
    .execute(&*state.apalis_pool)
    .await
    .map_err(|e| AppError::Internal(e.to_string()))?;

    if result.rows_affected() == 0 {
        return Err(AppError::NotFound(
            "Task not found or not in failed/killed status".into(),
        ));
    }

    Ok(Json(serde_json::json!({
        "message": "Task queued for retry",
        "task_id": task_id,
    })))
}
```
